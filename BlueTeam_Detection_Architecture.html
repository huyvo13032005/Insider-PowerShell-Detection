§0 Khái Niệm Nền Tảng
🎭

Insider Threat

Mối đe dọa đến từ người dùng hợp lệ nội bộ — nhân viên, contractor, cựu nhân viên — sử dụng quyền truy cập được cấp phép để gây hại. Không cần malware. Không cần exploit. Chỉ cần credential hợp lệ + công cụ hệ thống. Đây là lý do tại sao Insider Threat là khó phát hiện nhất: hành vi nhìn "bình thường".

🔧

Living-off-the-Land (LOLBins)

Kỹ thuật attacker chỉ sử dụng công cụ hệ thống hợp pháp (PowerShell, cmd, reg, certutil, mshta...) để thực hiện attack chain. Không cần upload malware → bypass AV/EDR. LOLBins được dùng vì: đã có sẵn trên Windows, có chữ ký Microsoft, được trust bởi whitelist.

⚡

PowerShell Abuse

PowerShell là Swiss Army Knife của attacker: file enumeration, network request, registry modification, archive creation đều có thể làm qua PS. -ExecutionPolicy Bypass bỏ qua restriction. -EncodedCommand obfuscate command. Đây là lý do PowerShell Logging (Script Block + Command Line) là BẮT BUỘC trong môi trường doanh nghiệp.

🔬

Detection Engineering

Quá trình thiết kế, xây dựng và tuning các detection rule có tính kỹ thuật cao. Không chỉ là "bật Sysmon lên". Detection Engineering bao gồm: phân tích attacker TTP, mapping MITRE, xác định log source, viết rule, đo FP/FN rate, và liên tục cải thiện. Mục tiêu: signal/noise ratio cao nhất có thể.

🔗

Correlation Rule vs Single Rule

Single rule: phát hiện một event đơn lẻ (ví dụ: PowerShell chạy). FP cao vì mỗi event riêng lẻ có thể legitimate. Correlation rule: yêu cầu một chuỗi event theo thứ tự trong time window. FP thấp hơn nhiều vì xác suất ngẫu nhiên có đủ chuỗi events là rất thấp. Đây là nền tảng của detection behavioral.

🧠

Behavioral Detection

Thay vì detect signature (hash, IOC cụ thể), behavioral detection focus vào pattern hành vi: "ai làm gì, theo thứ tự nào, trong bao lâu". Ưu điểm: phát hiện được zero-day và unknown attacker. Nhược: cần baseline để định nghĩa "bình thường". Đây là hướng phát triển của modern SIEM/XDR.

🚨

SIEM Alert Pipeline

Log source → collect (Elastic Agent) → parse (ECS normalization) → index (Elasticsearch) → evaluate (detection rule chạy theo interval) → match → alert sinh ra (alert document) → notify (webhook/email) → ticket (OS Ticket). Mỗi bước đều có thể gây mất alert nếu cấu hình sai.

📋

Elastic ECS (Common Schema)

Elastic Common Schema là chuẩn field name thống nhất. process.name, process.command_line, file.path, registry.path, network.direction luôn nhất quán dù log đến từ Sysmon hay Endpoint Agent. Detection rule viết theo ECS → portable, không phụ thuộc vào log source cụ thể.

§3 Detection Rules Tối Ưu
🔴 Rule 1 — PowerShell Suspicious Execution (Execution Phase)
T1059.001 HIGH risk_score: 65 EQL + KQL
JSONRule 1 — PowerShell Execution Detection (Optimized)
{
  "name": "Suspicious PowerShell Execution — ExecutionPolicy Bypass",
  "description": "Detects PowerShell launched with bypass flags commonly used by insiders and attackers. Filters known admin tools to reduce FP.",
  "risk_score": 65,
  "severity": "high",
  "type": "eql",
  "language": "eql",
  "index": [
    "logs-endpoint.events.process*",
    "logs-windows.sysmon_operational-*",
    "winlogbeat-*"
  ],
  "query": """
    process where event.type == "start"
      and process.name like~ "powershell.exe"
      and (
        process.command_line like~ "*-ExecutionPolicy Bypass*"
        or process.command_line like~ "*-ExecutionPolicy Unrestricted*"
        or process.command_line like~ "*-ep bypass*"
        or process.command_line like~ "*-enc *"
        or process.command_line like~ "*-EncodedCommand*"
        or process.command_line like~ "*-noprofile*"
        or process.command_line like~ "*-windowstyle hidden*"
        or process.command_line like~ "*-w hidden*"
      )
      and not (
        process.parent.name like~ "msiexec.exe"
        or process.parent.name like~ "svchost.exe"
        or process.parent.name like~ "services.exe"
        or process.parent.executable like~ "*\\\\WindowsPowerShell\\\\v1.0\\\\powershell.exe"
        or process.args like~ "*Microsoft*Update*"
      )
  """,
  "interval": "5m",
  "from": "now-6m",
  "max_signals": 100,
  "tags": [
    "Execution", "MITRE T1059.001",
    "LOLBins", "PowerShell", "Insider Threat"
  ],
  "threat": [{
    "framework": "MITRE ATT&CK",
    "tactic": { "id": "TA0002", "name": "Execution" },
    "technique": [{ "id": "T1059", "name": "Command and Scripting Interpreter",
      "subtechnique": [{ "id": "T1059.001", "name": "PowerShell" }]
    }]
  }]
}

🎯 Mục đích: Phát hiện PowerShell được khởi chạy với các flags dùng để bypass execution policy hoặc ẩn hoạt động. Đây là dấu hiệu attacker đang sử dụng LOLBins technique.

📈 Cải tiến so với bản gốc: Rule gốc chỉ dùng KQL process.name : "powershell.exe" AND process.command_line : "ExecutionPolicy" — quá đơn giản, FP cao. Bản tối ưu dùng EQL với event.type == "start", thêm nhiều bypass variants (-ep bypass, -enc, -w hidden), và exclude known legitimate parents.

⚠️ False Positive khả năng: IT admin scripts dùng -ExecutionPolicy Bypass để chạy maintenance scripts. Mitigation: thêm user.name whitelist hoặc host.name whitelist cho admin machines.

📁 Index: Ưu tiên logs-endpoint.events.process* (Elastic Endpoint) vì đã normalize theo ECS. Fallback sang logs-windows.sysmon_operational-*.

🟡 Rule 2 — Sensitive Directory Enumeration (Discovery Phase)
T1083 MEDIUM risk_score: 45 Context-aware
JSONRule 2 — Directory Discovery via PowerShell (Optimized)
{
  "name": "PowerShell Sensitive Directory Enumeration",
  "description": "Detects PowerShell Get-ChildItem targeting sensitive directories. Standalone FP is high; most useful in correlation context.",
  "risk_score": 45,
  "severity": "medium",
  "type": "eql",
  "language": "eql",
  "index": [
    "logs-endpoint.events.process*",
    "logs-windows.sysmon_operational-*"
  ],
  "query": """
    process where event.type == "start"
      and process.name like~ "powershell.exe"
      and process.command_line like~ "*Get-ChildItem*"
      and (
        process.command_line like~ "*-Recurse*"
        or process.command_line like~ "*-r *"
      )
      and (
        process.command_line like~ "*C:\\\\Users\\\\*"
        or process.command_line like~ "*C:\\\\Projects*"
        or process.command_line like~ "*C:\\\\Confidential*"
        or process.command_line like~ "*\\\\\\\\*"
        or process.command_line like~ "*AppData*"
        or process.command_line like~ "*.txt*"
        or process.command_line like~ "*.docx*"
        or process.command_line like~ "*.xlsx*"
        or process.command_line like~ "*.pdf*"
      )
      and not process.parent.name like~ "explorer.exe"
  """,
  "interval": "5m",
  "from": "now-6m",
  "tags": [
    "Discovery", "MITRE T1083",
    "PowerShell", "File Enumeration"
  ],
  "note": "Low-fidelity standalone. Most effective as part of correlation sequence. Consider adjusting risk_score based on user role."
}

🎯 Mục đích: Phát hiện việc liệt kê thư mục nhạy cảm qua PowerShell với -Recurse. Standalone signal yếu, nhưng trong chuỗi correlation thì rất có giá trị.

📈 Cải tiến: Rule gốc dùng Get-ChildItem OR dir — cực kỳ FP, hàng ngàn event mỗi ngày. Bản tối ưu yêu cầu cả Get-ChildItem AND -Recurse AND target là sensitive path (C:\Users, C:\Projects, shares). Exclude explorer.exe parent (file manager normal operation).

⚠️ False Positive: Trung bình. Dev/Ops script hợp lệ cũng recurse qua C:\Users. Giảm FP bằng cách kết hợp với Rule 1 trong correlation.

🔵 Rule 3 — Data Staging via Archive Creation (Collection Phase)
T1560.001 HIGH risk_score: 70 EID 11 + EID 1
JSONRule 3 — Archive Creation in Suspicious Location (Optimized)
{
  "name": "PowerShell Archive Creation in Staging Location",
  "description": "Detects Compress-Archive writing zip files to world-writable or temp locations. Higher fidelity than command-line only detection.",
  "risk_score": 70,
  "severity": "high",
  "type": "eql",
  "language": "eql",
  "index": [
    "logs-endpoint.events.process*",
    "logs-endpoint.events.file*",
    "logs-windows.sysmon_operational-*"
  ],
  "query": """
    sequence by host.name, user.name with maxspan=2m
      [process where event.type == "start"
        and process.name like~ "powershell.exe"
        and process.command_line like~ "*Compress-Archive*"
        and (
          process.command_line like~ "*C:\\\\Users\\\\Public\\\\*"
          or process.command_line like~ "*C:\\\\Temp\\\\*"
          or process.command_line like~ "*C:\\\\Windows\\\\Temp\\\\*"
          or process.command_line like~ "*%TEMP%*"
          or process.command_line like~ "*%APPDATA%*"
        )
      ]
      [file where event.type in ("creation", "change")
        and file.extension in ("zip", "7z", "rar", "tar", "gz")
        and (
          file.path like~ "*\\\\Public\\\\*"
          or file.path like~ "*\\\\Temp\\\\*"
          or file.path like~ "*\\\\AppData\\\\Roaming\\\\*"
        )
      ]
  """,
  "interval": "5m",
  "from": "now-6m",
  "tags": [
    "Collection", "MITRE T1560.001",
    "Data Staging", "Archive"
  ]
}

🎯 Mục đích: Phát hiện tạo archive trong các staging location phổ biến của attacker. Kết hợp cả process event (EID 1) VÀ file creation event (EID 11) để tăng fidelity.

📈 Cải tiến: Rule gốc chỉ match process.command_line : "Compress-Archive" — người dùng bình thường cũng zip files. Bản tối ưu yêu cầu destination phải là staging path (C:\Public, C:\Temp) VÀ xác nhận file .zip thực sự được tạo (EID 11). Sử dụng mini-sequence theo user.name trong 2 phút.

⚠️ False Positive: Thấp-Trung bình. Người dùng hiếm khi zip vào C:\Public. IT tools đôi khi backup vào Temp — cần whitelist process.parent nếu cần.

🎯 Mục đích: Phát hiện Run key chứa script interpreter — dấu hiệu rõ ràng của persistence mechanism. Không phải software hợp lệ nào cũng cần powershell.exe trong Run key.

📈 Cải tiến: Rule gốc dùng registry.path : "*\\Run*" — quá broad, mọi phần mềm cài đặt đều write vào Run key. Bản tối ưu lọc theo registry.data.strings (nội dung value) phải chứa interpreter (powershell, cmd, wscript). Exclude known legitimate apps. Sử dụng EID 13 (Registry Value Set) thay vì chỉ dựa vào process.command_line.

⚠️ False Positive: Thấp. Run key với powershell.exe là rất bất thường trong môi trường doanh nghiệp. Cần tuning theo environment cụ thể (add legitimate apps vào exclusion list).

⚠️ False Positive: Trung bình. Nhiều automation scripts dùng PS để call APIs. Giảm FP bằng cách yêu cầu correlation với Rule 3 (archive) trước.

🎯 Mục đích: Phát hiện xóa đệ quy nhiều file quan trọng qua PowerShell. Đây là hành vi Data Destruction đặc trưng của insider angry/departing.

📈 Cải tiến critical: Rule gốc dùng process.command_line : "Remove-Item" OR "del" — sai nghiêm trọng. Remove-Item và del là lệnh cực kỳ phổ biến, gây hàng nghìn FP/ngày. Bản tối ưu: (1) yêu cầu -Recurse flag, (2) target phải là user/project directory, (3) xác nhận file deletion thực sự xảy ra qua EID 23, (4) check file extension là file quan trọng.

⚠️ False Positive: Thấp khi yêu cầu đủ điều kiện. Cần Sysmon config bật EID 23.

§4 Correlation Rule — Full Attack Chain (EQL Sequence)
🚨 Lỗi trong Correlation Rule gốc:
  • Wildcard trong EQL registry cần dùng like~ syntax, không phải KQL-style : "*...*"
  • Thiếu by user.name — chỉ group theo host.name dễ bị bypass nếu attacker switch user
  • maxspan=30m quá ngắn — insider attack thực tế có thể kéo dài vài giờ
  • Rule gốc không sinh alert vì index pattern logs-* không match logs-windows.sysmon_operational-*
  • Không có event.type filter → match cả update/create/delete events lẫn nhau

🔍 Tại sao correlation phát hiện tốt hơn single rule? Single rule sinh alert cho mỗi bước (Execution → hàng trăm alerts/ngày). Correlation chỉ sinh alert khi TẤT CẢ 6 bước xảy ra — số lượng alert ít hơn nhiều, nhưng confidence cực cao. SOC analyst không bị "alert fatigue".

§5 Tối Ưu Alert Trên Elastic SIEM
❓ Tại Sao Rule Hiện Tại Không Sinh Alert?
Nguyên nhân #1 (Index Pattern): Rule gốc dùng "index": ["logs-*", "winlogbeat-*"] nhưng Elastic Agent với Sysmon integration ghi log vào logs-windows.sysmon_operational-* hoặc logs-endpoint.events.*. Pattern logs-* không match data stream logs-windows.sysmon_operational-default nếu lifecycle policy tạo riêng.
Nguyên nhân #2 (ECS Field Mapping): Rule dùng process.command_line nhưng data thực tế từ Sysmon có thể ở field winlog.event_data.CommandLine nếu chưa được normalize. Cần verify trong Discover trước khi viết rule.
Nguyên nhân #3 (KQL vs EQL): Rule gốc dùng type: "query" với language: "kuery" nhưng KQL wildcard : "*..\\.Run*" cho registry path có thể fail vì backslash escaping. EQL với like~ xử lý tốt hơn.
Nguyên nhân #4 (lookback window): Rule dùng "from": "now-5m" với "interval": "5m". Nếu rule chạy trễ 1 giây, có thể có event "rơi" vào khoảng trống. Best practice: from: now-6m với interval: 5m (1 phút overlap).
⚙️ Best Practice Tuning Parameters
Index Pattern
Dùng logs-endpoint.events.process* (Elastic Agent) HOẶC logs-windows.sysmon_operational-* (Winlogbeat). Verify trong Discover tab trước. Không dùng logs-* vì quá broad → performance issue.
Interval
5m cho most rules. Correlation rule nên dùng 5m với from: now-2h. Không dùng interval ngắn hơn 1m (performance). Không dùng quá 15m (detection lag quá lớn).
Lookback Window
Luôn đặt from = interval + 1 phút buffer. Ví dụ: interval=5m → from=now-6m. Correlation rule: from phải = maxspan. Tránh gap giữa các lần chạy rule.
max_signals
Single rules: 100. Correlation rule: 5-10 (rất ít legitimate alerts). Nếu rule sinh quá nhiều alert → signal là bị nhiễu, cần tuning thêm.
ECS Normalization
Verify fields tồn tại: trong Kibana Discover → filter theo index → inspect document. Đảm bảo process.command_line, registry.data.strings, network.direction được populate. Nếu không có → thêm Elastic Integration pipeline.
Rule Type
Ưu tiên EQL cho tất cả rules (syntax mạnh hơn, hỗ trợ sequence, wildcard chuẩn hơn). KQL chỉ dùng cho simple ad-hoc queries. Threshold rules phù hợp cho anomaly detection (e.g., >50 file deletions trong 5 phút).
Actions / Notifications
Kết nối webhook → OS Ticket API để tự động tạo ticket khi alert sinh ra. Cấu hình trong Rule → Actions → Add Action → Webhook. Dùng Mustache template để include host.name, user.name, rule.name trong ticket body.
🔧 Cách Sửa Để Alert Hoạt Động — Checklist

Step 1: Vào Kibana → Discover → chọn index logs-windows.sysmon_operational-* → verify có event.code, process.name, process.command_line fields.

Step 2: Vào Security → Rules → chỉnh index pattern của tất cả rules sang index đúng (xem §2). Xóa winlogbeat-* nếu không dùng Winlogbeat.

Step 3: Chuyển tất cả rules từ type: "query" + language: "kuery" sang type: "eql" + language: "eql" với syntax đúng (như §3).

Step 4: Điều chỉnh "from": "now-6m" cho tất cả 5m-interval rules. Correlation rule: "from": "now-2h".

Step 5: Enable Sysmon EID 23 trong sysmonconfig.xml: thêm <FileDelete onmatch="include"> section. Reinstall Sysmon với config mới.

Step 6: Test manual: chạy lại attack script → vào Security Alerts → check xem alert có xuất hiện không. Nếu không → check rule execution log (Rule → Last Response).

Lỗi phổ biến: Rule enabled nhưng "Last Response: No results" → data không vào index đúng. Rule "Partial Failure" → EQL syntax error. Rule "Failed" → index permission issue.

§6 Telemetry Flow — End-to-End
💻
Windows 10 Host (ituser)
Insider chạy PowerShell → thực hiện attack chain. Mọi action tạo ra Windows Event.
👁️
Sysmon (System Monitor)

📡

🔍
Elasticsearch (Index)
Index documents vào data stream. ILM (Index Lifecycle Management) quản lý retention. Mapping tự động theo ECS schema. Sẵn sàng để query.
⚡
Elastic SIEM Detection Engine
Rule scheduler chạy mỗi 5 phút. Query Elasticsearch với EQL/KQL. Match → tạo alert document trong .alerts-security.alerts-default index. Trigger configured actions (webhook, email).

Alert → OS Ticket
Webhook action gọi OS Ticket API → tự động tạo ticket với severity, host, user, MITRE tags. SOC analyst nhận ticket → investigate → contain → eradicate → recover.

🎯 Khuyến Nghị Bổ Sung (Beyond Lab Scope)
📊 User Behavior Analytics (UBA)

Elastic ML jobs để detect anomalous behavior: user first time accessing path, unusual hours login, spike in file access count. Dùng Anomaly Detection jobs: v3_rare_process_by_host, v3_windows_rare_user_runas_event.

🌐 DNS Logging

Bật Sysmon EID 22 (DNS Query) để detect exfiltration over DNS hoặc DGA domains. Thêm vào correlation: DNS query từ powershell.exe to external domain ngay trước network connection.

🔐 Privileged Access Monitoring

Audit Windows Security Event 4648 (explicit credential use), 4624 (logon), 4634 (logoff) để track lateral movement attempt. Insider đôi khi cố escalate privilege.

🚫 Constrained Language Mode

Enable PowerShell Constrained Language Mode qua AppLocker/WDAC để prevent advanced PS abuse. Attacker bypass CLM thường visible trong Script Block logs (WinEvent 4104).

📦 DLP Integration

Kết hợp SIEM alert với Data Loss Prevention solution (Microsoft Purview, Symantec DLP) để detect large file transfer, USB copy, cloud upload. Correlation giữa DLP alert + SIEM rule = highest confidence.

🔄 Automated Response (SOAR)

Kết nối Elastic với SOAR (Tines, Splunk SOAR, Palo Alto XSOAR). Critical alert → tự động isolate host (Elastic Endpoint response action), disable AD account, create OS Ticket, notify manager. Giảm MTTR từ giờ xuống phút.
